# CHAPTER 3: REQUIREMENT ANALYSIS

3.a. Requirement Gathering Methodology
The requirement gathering process for the AquaSense project was a rigorous four-phase operation designed to capture the complexities of urban water management. In the first phase, known as Stakeholder Discovery, we engaged in detailed consultations with residential facility managers at DKTE University. These discussions highlighted a critical lack of granular data, revealing that while total water consumption was known, the specific points of wastage—such as leaks in utility lines or excessive bathroom usage—remained invisible. This established the primary functional requirement for area-wise telemetry. 

In the second phase, Market and Gap Analysis, we performed a comparative study of existing industrial smart-meters. We identified a recurring failure in current systems: they are "reactive" rather than "proactive." Most systems only provide historical data, whereas AquaSense was conceptualized to provide real-time, millisecond-accurate feedback. This led to the prioritization of a software-defined actuation layer. The third phase involved Software Simulation Feasibility Studies. Given that this implementation focuses on the software layer, we gathered flow-rate benchmarks from domestic standards (ranging from 6 to 12 liters per minute) and used these to program our virtual sensors. Finally, in the Validation phase, we cross-referenced our gathered requirements against the DKTE Mini Project guidelines to ensure the project met all academic and engineering standards for a software-only submission.

3.b. Requirement Specification (SRS)

3.b.1 Introduction Purpose
The primary purpose of this Software Requirements Specification (SRS) is to provide a comprehensive and definitive technical description of the AquaSense software architecture. It is intended to serve as a baseline for the development team and as a formal verification document for the project evaluators. This document outlines the functional capabilities, performance constraints, and user interface standards that the system must adhere to. By formalizing these requirements, we ensure that the integration between the C++ Hardware Simulator and the Python Flask Logic Gateway is seamless, reliable, and mathematically accurate. This SRS effectively acts as the technical contract for the project, defining the "Success Criteria" for the final software-only validation.

3.b.2 Scope of the System
The scope of the AquaSense ecosystem is strictly defined as a high-fidelity digital twin for intelligent water resource governance. Horizontally, the system is designed to monitor three distinct usage zones: the Kitchen, the Bathroom, and the Garden/Plantation area. It also monitors a centralized virtual storage tank. Vertically, the system encompasses the entire data lifecycle, from the emulation of raw sensor interrupts at the perception layer to the generation of complex time-series analytics at the presentation layer. While the system is robust, it is currently scoped for small-to-medium scale institutional deployments. It excludes physical hydraulic engineering, high-voltage electrical wiring for pumps, and the integration of commercial billing or third-party payment gateways, which are reserved for future future industrial iterations.

3.b.4 Detailed System Features
The system is comprised of five core functional features that drive its intelligence. Feature 1 (Telemetry Ingestion) allows the backend to process high-frequency JSON payloads from multiple simulated sources simultaneously without data collision. Feature 2 (Multi-Area Visualization) provides the end-user with a granular view of their water health, using dynamic progress rings and line charts to distinguish between different zones. Feature 3 (Adaptive Aeration Logic) is the system’s primary conservation tool; it calculates the exact percentage of flow reduction required based on the current water level in the storage tank. Feature 4 (Fail-Safe Anomaly Mitigation) acts as the system’s security guard, detecting high-flow spikes that indicate a pipe burst and triggering an immediate emergency shutdown. Feature 5 (Distributed Alert Handshake) ensures the user is always informed by bridging the gap between the internal server logic and the external Telegram Bot API for instant notifications.

3.b.5 Non-Functional Requirements
Beyond its core features, AquaSense must meet several high-performance non-functional standards. In terms of Performance and Latency, the system is required to process any telemetry event and update the dashboard in less than 1.5 seconds to ensure real-time responsiveness. Reliability is ensured through atomic write operations to the CSV database, preventing file corruption during power failures. Scalability is a core architectural goal; the backend is designed to support up to 50 simulated zones with minimal impact on CPU utilization. Availability is maintained through the use of non-blocking I/O, ensuring that the heavy task of data ingestion does not slow down the user’s experience on the web dashboard. Security, while limited in this prototype, is handled via unique API key validation for all simulator-to-server handshakes.

3.b.6 User Interface (UI) Design Specification
The user interface for AquaSense is designed to be an "Executive Command Center," utilizing the modern Glassmorphism aesthetic. This involves the use of deep navy backgrounds paired with translucent, frosted-glass panels that provide a high-end, futuristic feel. The UI utilizes a strict color-coded hierarchy: Cyan represents normal operational status, Yellow indicates that the system has entered an autonomous conservation mode, and Neon Red signifies an active anomaly or leak. Data visualization is handled via the Chart.js engine, which renders smooth, animated line graphs for all zones. The interface is built with a mobile-first philosophy, utilizing responsive CSS Grid and Flexbox layouts to ensure that the dashboard remains perfectly legible whether viewed on a 4K desktop monitor or a standard 6-inch mobile device.
